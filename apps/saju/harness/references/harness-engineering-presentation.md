# Harness Engineering 발표안

- 추천 스타일: Swiss International

- 핵심 톤: 쉽게, 단정하게, 그러나 실제 실행 예시는 구체적으로
- 컬러: 화이트 배경, 거의 검정 텍스트, 레드 포인트 1색
- 폰트: 제목은 Helvetica/Arial Bold, 본문은 Helvetica/Arial, 캡션은 모노스페이스

## Slide 1. Title

- 제목: 사주 서비스를 하네스 엔지니어링으로 만들자
- 부제: 프롬프트를 넘어서, AI와 일하는 환경을 설계하는 법
- 하단 캡션: Ask Dori

발표자 노트:
안녕하세요 에스크 도리의 도리입니다. 오늘은 하네스 엔지니어링이 무엇인지, 그리고 그걸 실제로 사주 서비스 제작에 어떻게 적용할 수 있는지 같이 보겠습니다.

## Slide 2. Why This Talk

- 댓글로 가장 많이 받은 질문
- "하네스 엔지니어링은 어떻게 하나요?"
- 오늘의 목표
- 개념 이해
- 실제 구조 보기
- 사주 서비스 예시로 실행 흐름 익히기

발표자 노트:
하네스 엔지니어링이라는 말을 쓰는 분은 늘고 있는데, 막상 쉽게 설명하는 자료는 많지 않았습니다. 그래서 오늘은 개념 설명으로 끝내지 않고, 실제로 어떻게 착수하는지까지 같이 가보겠습니다.

## Slide 3. Prompt Engineering vs Harness Engineering

- 프롬프트 엔지니어링
- 그때그때 잘 요청하는 기술
- 하네스 엔지니어링
- AI가 잘 일하게 만드는 환경 설계

발표자 노트:
우리는 보통 "이거 해줘"라고 자연어로 요청해왔고, 그걸 프롬프트 엔지니어링이라고 불렀습니다. 하네스 엔지니어링은 여기서 한 단계 더 가서, AI가 반복적으로 잘 일할 수 있도록 작업 환경 전체를 설계하는 접근입니다.

## Slide 4. Think of AI as a New Teammate

- AI는 도구이기도 하지만
- 동시에 새로 온 팀원처럼 다뤄야 한다
- 팀 규칙
- 결과물 기준
- 금지사항
- 작업 흐름

발표자 노트:
매번 처음부터 설명하기보다, 우리 팀은 이렇게 일한다는 규칙과 기준을 먼저 알려주는 편이 훨씬 일관된 결과를 만듭니다. 바쁜 팀장일수록 이 셋업이 중요합니다.

## Slide 5. What Harness Engineering Is

- 업무 위임을 위한 운영 설계
- 규칙
- 작업 구조
- 검증 방법
- 실행 흐름

발표자 노트:
하네스 엔지니어링은 LLM에게 일을 맡기기 위해, 규칙과 작업 구조, 검증 방식, 실행 루프를 미리 묶어두는 일입니다. 즉흥적으로 시키는 게 아니라, 일하는 체계를 만드는 것입니다.

## Slide 6. Four Core Components

- Constitution
- Task Structure
- Validation
- Agent Loop

발표자 노트:
헌법은 "우리 팀은 이렇게 일한다"는 규칙이고, 작업 구조는 무엇을 어떻게 만들지 정하는 지시서입니다. 검증은 결과가 맞는지 확인하는 기준이고, 실행 루프는 수정-검증-반복의 흐름입니다.

## Slide 7. More Than Four Documents

- 문서만 있으면 끝이 아니다
- 저장소가 필요하다
- 참고 자료가 필요하다
- rollback 장치가 필요하다

발표자 노트:
규칙과 작업 구조만으로는 부족합니다. 실제로 작업할 repository와 참고 자료, 그리고 실패했을 때 되돌릴 장치까지 있어야 AI가 자율적으로 움직일 수 있습니다.

## Slide 8. Two Levels of Harness

- Repository Harness
- 이 저장소 전체에서 어떻게 일할지
- Application Harness
- 이 앱이 어떤 제품이어야 하는지

발표자 노트:
하네스 엔지니어링은 크게 두 단계로 나눌 수 있습니다. 저장소 전체에 적용되는 공통 규칙이 있고, 각 앱이나 서비스에 맞는 세부 규칙이 있습니다.

## Slide 9. Why Start from the App

- 개발자는 보통 repository부터
- 일반 사용자는 보통 app부터 떠올린다
- 그래서 이번 강의는 귀납적으로 진행

발표자 노트:
개발자는 작업 환경부터 세팅하는 데 익숙하지만, 일반 사용자는 보통 만들고 싶은 앱이 먼저 떠오릅니다. 그래서 오늘은 애플리케이션 하네스를 먼저 생각하고, 이후 repository 하네스로 확장하는 흐름을 택합니다.

## Slide 10. Demo Target: Saju Service

- 교육용 규칙 기반 사주 서비스
- 생년월일시 입력
- 기초 사주 분석 결과 출력
- LLM API 없이도 구현 가능

발표자 노트:
최근에 지인이 사주 서비스를 만든 걸 보고 재미있게 봤고, 교육용 예제로도 적합하다고 느꼈습니다. 이번 예시는 아주 단순한 규칙 기반으로, 해석 엔진의 기본 구조만 보여주는 방향입니다.

## Slide 11. Starting Point

- OpenAI 하네스 엔지니어링 아티클 참고
- 표준 구조를 복사
- 우리 목적에 맞게 조정

발표자 노트:
처음부터 모든 구조를 혼자 발명할 필요는 없습니다. 표준적인 구조를 가져와서 내 작업 방식과 목적에 맞게 수정하면 됩니다.

## Slide 12. Example Repository Structure

- `harness/core/`
- 공통 문서, 역할, 워크플로우
- `apps/saju/harness/`
- 앱 전용 문서, 명세, 계획
- `src/`, `tests/`
- 실제 코드와 검증

발표자 노트:
repository 레벨에는 공통 규칙을 두고, app 레벨에는 사주 서비스에 맞는 문서를 둡니다. 그리고 코드와 테스트는 별도 폴더로 분리해서 실행 가능한 구조를 만듭니다.

## Slide 13. Constitution Matters Most

- 인간이 가장 많이 개입해야 하는 부분
- 왜 이 저장소를 만드는가
- 무엇을 지키고 무엇을 금지하는가

발표자 노트:
AI가 협업할 때 가장 중요한 건 헌법입니다. 교육용이라면 간결해야 하고, 안전해야 하고, 초보자가 이해할 수 있어야 합니다. 이 가치 판단은 결국 사람이 잡아줘야 합니다.

## Slide 14. Sample Constitution

- 간결성
- 검증 가능성
- 안전성
- 문서 없이 기능 추가 금지
- 개인정보 로그 저장 금지

발표자 노트:
예를 들어 저의 저장소 헌법은 간결성, 검증 가능성, 안전성을 핵심 가치로 두고 있습니다. 이런 원칙이 있어야 이후 문서와 코드, 테스트 판단이 흔들리지 않습니다.

## Slide 15. Review for Ambiguity

- 모호한 표현을 제거해야 한다
- 숫자는 가능하면 숫자로
- 역할과 handoff 조건은 명확하게

발표자 노트:
문서를 AI와 같이 쓰다 보면 그럴듯하지만 애매한 표현이 많이 들어갑니다. 예를 들어 "한두 번 수정 후" 같은 표현은 기준이 너무 약합니다. 이런 지점은 사람이 보고 더 명확하게 바꾸는 게 중요합니다.

## Slide 16. Cleanup and Rollback

- 자율 cleanup agent
- snapshot restore
- git checkpoint
- branch rollback

발표자 노트:
AI에게 자율성을 줄수록 잘못된 수정에서 빠르게 되돌릴 수 있는 안전장치가 필요합니다. 로컬 cleanup과 git rollback을 2층 구조로 두면 훨씬 안전합니다.

## Slide 17. Why Git Is Essential

- 파일 단위 복구만으로는 부족
- 작업 단위 전체를 되돌려야 할 때가 있다
- git은 강한 rollback 레이어다

발표자 노트:
cleanup snapshot은 일부 파일 복구에는 좋지만, 작업 전체를 되돌리기에는 한계가 있습니다. 그래서 git branch, checkpoint commit, restore 같은 기능이 꼭 필요합니다.

## Slide 18. Execution Loop

- 문서 작성
- 코드 구현
- 테스트
- 코드 리뷰
- 다시 수정

발표자 노트:
하네스 엔지니어링은 문서를 만드는 것으로 끝나지 않습니다. 문서대로 구현하고, 실제로 실행해 보고, 테스트를 추가하고, 다시 고치는 루프를 돌려야 진짜로 살아 있는 시스템이 됩니다.

## Slide 19. Human in the Loop

- AI가 만든 문서는 반드시 사람이 읽고 승인
- 방임도 안 되고
- 미시적 지시도 안 된다
- 합의 기반 위임이 핵심

발표자 노트:
AI가 만든 결과를 그냥 믿고 넘기면 안 됩니다. 하지만 모든 걸 하나하나 지시하는 것도 비효율적입니다. 팀장은 목표와 기준을 분명히 하고, AI와 합의하면서 자율성을 주는 쪽으로 가야 합니다.

## Slide 20. Final Message

- 프롬프트를 잘 쓰는 것에서 끝나지 말자
- AI가 잘 일하는 환경까지 설계하자
- 하네스 엔지니어링은 기술이자 운영 방식이다

발표자 노트:
정리하면, 하네스 엔지니어링은 AI에게 일을 던지는 방식이 아니라, AI가 반복적으로 좋은 결과를 내도록 일하는 환경을 설계하는 작업입니다. 저는 이것이 단순한 테크닉을 넘어, 경영과 운영에 가까운 사고방식이라고 생각합니다.
