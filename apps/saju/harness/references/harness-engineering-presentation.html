Today

개념부터 시작해서 repository harness, application harness, 그리고 실제 실행 루프까지 한 번에 본다.

“AI에게 일을 시키는 것”이 아니라 “AI가 잘 일하게 만드는 환경”을 설계하는 이야기다.Opening thesis
안녕하세요 에스크 도리의 도리입니다. 오늘은 하네스 엔지니어링이 무엇인지, 그리고 그걸 실제로 사주 서비스 제작에 어떻게 적용할 수 있는지 같이 보겠습니다.

Concept

Structure

문서, 저장소, 검증 기준이 어떻게 연결되는지 보여준다.

Practice

사주 서비스 예시로 실제 적용 흐름까지 따라간다.

하네스 엔지니어링이라는 말을 쓰는 분은 늘고 있는데, 막상 쉽게 설명하는 자료는 많지 않았습니다. 그래서 오늘은 개념 설명으로 끝내지 않고, 실제로 어떻게 착수하는지까지 같이 가보겠습니다.
03
Comparison
Request vs environment
출발점 비교

프롬프트 엔지니어링에서 하네스 엔지니어링으로

Prompt Engineering

  • 그때그때 요청을 잘 쓰는 기술
  • 모델에게 한 번의 작업을 부탁하는 방식
  • 결과 품질이 매 요청에 크게 좌우된다

Harness Engineering

  • AI가 잘 일하게 만드는 환경 설계
  • 규칙, 구조, 검증, 루프를 미리 합의한다
  • 반복 가능한 품질과 일관성을 노린다
우리는 보통 "이거 해줘"라고 자연어로 요청해왔고, 그걸 프롬프트 엔지니어링이라고 불렀습니다. 하네스 엔지니어링은 여기서 한 단계 더 가서, AI가 반복적으로 잘 일할 수 있도록 작업 환경 전체를 설계하는 접근입니다.
04

관점 전환

AI를 도구가 아니라 새로 온 팀원으로 본다면?

“우리 회사는 이렇게 일해. 이런 결과물을 원해. 이런 건 꼭 지켜. 이런 건 하지 마.”팀 규칙을 먼저 공유하는 방식
  • 팀 규칙을 알려주면 매번 처음부터 설명할 필요가 없다.
  • 결과물 기준과 금지사항이 일관성을 만든다.
  • 바쁜 팀장일수록 환경 설계가 효율을 만든다.
매번 처음부터 설명하기보다, 우리 팀은 이렇게 일한다는 규칙과 기준을 먼저 알려주는 편이 훨씬 일관된 결과를 만듭니다. 바쁜 팀장일수록 이 셋업이 중요합니다.

Constitution

07
Reality check
Docs are not enough
실전에서 빠지기 쉬운 것

문서만 있다고 AI가 잘 일하는 건 아니다

  • Repository가 있어야 어디에서 일할지 정해진다.
  • Reference가 있어야 복잡한 판단을 보조할 수 있다.
  • Rollback이 있어야 자율적으로 움직이다가도 안전하게 돌아온다.
규칙 + 구조 + 검증 + 루프 + 저장소 + 참고 자료 + 복구 장치까지 있어야 오른팔처럼 일한다.Operational completeness
규칙과 작업 구조만으로는 부족합니다. 실제로 작업할 repository와 참고 자료, 그리고 실패했을 때 되돌릴 장치까지 있어야 AI가 자율적으로 움직일 수 있습니다.
08
Two levels
Repository and app
큰 그림

하네스는 두 단계로 나뉜다

Repository Harness

이 저장소 전체에서 어떻게 일할지 정한다. 문서 위치, 공통 규칙, 역할, 린트/테스트/CI 관점, handoff 흐름이 여기에 들어간다.

Application Harness

특정 앱이 어떤 기능과 사용자 경험을 가져야 하는지 정의한다. 도메인 규칙, 명세, 피드백 구조, 앱 전용 에이전트 루프가 여기에 들어간다.

하네스 엔지니어링은 크게 두 단계로 나눌 수 있습니다. 저장소 전체에 적용되는 공통 규칙이 있고, 각 앱이나 서비스에 맞는 세부 규칙이 있습니다.

Teaching choice
Inductive route
왜 이 순서인가

개발자는 repository부터, 일반 사용자는 app부터 떠올린다

개발자에게 익숙한 방식

  • 작업 환경부터 세팅
  • 공통 규칙부터 먼저 설계
  • 그 다음 앱을 구현

이번 강의의 방식

  • 만들고 싶은 앱을 먼저 잡는다
  • 앱 하네스를 먼저 명확히 한다
  • 그 후 repository 하네스로 확장한다
개발자는 작업 환경부터 세팅하는 데 익숙하지만, 일반 사용자는 보통 만들고 싶은 앱이 먼저 떠오릅니다. 그래서 오늘은 애플리케이션 하네스를 먼저 생각하고, 이후 repository 하네스로 확장하는 흐름을 택합니다.

헌법은 AI 협업의 방향을 고정하는 문서다

Purpose

왜 이 저장소를 만드는지 분명하게 적는다.

Done

AI가 협업할 때 가장 중요한 건 헌법입니다. 교육용이라면 간결해야 하고, 안전해야 하고, 초보자가 이해할 수 있어야 합니다. 이 가치 판단은 결국 사람이 잡아줘야 합니다.
14
Example
Clear non-negotiables
예시 원칙

이 저장소의 헌법이 말해주는 것

간결성

복잡하면 쪼개고, 불필요하면 제거한다.

검증 가능성

문서, 코드, 테스트가 같은 결과를 확인해야 한다.

안전성

개인정보와 입력 데이터는 기본적으로 보호 대상이다.

비가역 금지

예를 들어 저의 저장소 헌법은 간결성, 검증 가능성, 안전성을 핵심 가치로 두고 있습니다. 이런 원칙이 있어야 이후 문서와 코드, 테스트 판단이 흔들리지 않습니다.
15
Tighten the rules
Remove vagueness
문서 다듬기

AI와 협업할수록 모호한 표현을 줄여야 한다

Before

  • “한두 번 수정 후”
  • “적당히 검토”
  • “필요하면 롤백”

After

  • “최대 5회까지 자율 수정 시도 후 기록”
  • “handoff 조건을 문서로 남긴다”
  • “rollback trigger를 숫자 기준으로 정의”
문서를 AI와 같이 쓰다 보면 그럴듯하지만 애매한 표현이 많이 들어갑니다. 예를 들어 "한두 번 수정 후" 같은 표현은 기준이 너무 약합니다. 이런 지점은 사람이 보고 더 명확하게 바꾸는 게 중요합니다.
16
Safety layer
Autonomy needs recovery
안전장치

AI의 자율성이 커질수록 cleanup과 rollback도 강해져야 한다

빠른 로컬 cleanup

temp file 정리, debug code 제거, 프로세스 종료, snapshot 저장 같은 국소 복구를 담당한다.

강한 git rollback

branch, checkpoint commit, restore를 통해 작업 단위 전체를 되돌릴 수 있게 만든다.

AI에게 자율성을 줄수록 잘못된 수정에서 빠르게 되돌릴 수 있는 안전장치가 필요합니다. 로컬 cleanup과 git rollback을 2층 구조로 두면 훨씬 안전합니다.
17
Rollback layer
Checkpoint thinking
왜 git이 필요한가

파일 단위 복구만으로는 작업 전체를 지킬 수 없다

  • cleanup snapshot은 일부 파일 복구에는 좋지만 범위가 좁다.
  • git은 작업 상태 전체를 기준으로 복구할 수 있다.
  • AI가 자율적으로 수정할수록 checkpoint가 더 중요해진다.
snapshot rollback = 국소 복구 / git rollback = 작업 단위 복구Two-layer recovery model
cleanup snapshot은 일부 파일 복구에는 좋지만, 작업 전체를 되돌리기에는 한계가 있습니다. 그래서 git branch, checkpoint commit, restore 같은 기능이 꼭 필요합니다.
18
Execution
Docs to code
실행 루프

문서만 만드는 게 아니라 계속 돌려야 한다

Step 1

문서 작성

헌법, 명세, 계획, 참고 자료를 만든다.

Step 2

코드 구현

문서 기준으로 가장 단순한 구현을 만든다.

Step 3

테스트

문서 기준으로 실제 동작을 검증한다.

Step 4

리뷰

보안, 안정성, 구조를 다시 본다.

Step 5

반복

수정하고 재검증하며 기준 통과까지 간다.

하네스 엔지니어링은 문서를 만드는 것으로 끝나지 않습니다. 문서대로 구현하고, 실제로 실행해 보고, 테스트를 추가하고, 다시 고치는 루프를 돌려야 진짜로 살아 있는 시스템이 됩니다.
19
Leadership
Not neglect, not micromanage
중요한 태도

방임도 아니고, 일일이 지시도 아닌 합의 기반 위임

Goal

수요자인 내가 원하는 결과의 상을 먼저 명확히 가진다.

Agreement

AI와 지식을 나누며 목표와 기준을 함께 구체화한다.

Review

AI가 만든 결과는 사람이 읽고 승인하면서 다듬는다.

AI가 만든 결과를 그냥 믿고 넘기면 안 됩니다. 하지만 모든 걸 하나하나 지시하는 것도 비효율적입니다. 팀장은 목표와 기준을 분명히 하고, AI와 합의하면서 자율성을 주는 쪽으로 가야 합니다.
20
Closing
Design the work environment
마무리

프롬프트를 잘 쓰는 것에서 끝나지 말자

AI에게 일을 던지는 대신, AI가 반복적으로 좋은 결과를 내는 환경을 설계하자.Harness engineering as an operating philosophy
  • 하네스 엔지니어링은 기술이면서 동시에 운영 방식이다.
  • 문서와 코드와 검증과 rollback이 하나의 시스템으로 이어져야 한다.
  • 좋은 팀장은 AI에게도 좋은 일하는 환경을 만들어준다.
정리하면, 하네스 엔지니어링은 AI에게 일을 던지는 방식이 아니라, AI가 반복적으로 좋은 결과를 내도록 일하는 환경을 설계하는 작업입니다. 저는 이것이 단순한 테크닉을 넘어, 경영과 운영에 가까운 사고방식이라고 생각합니다.
